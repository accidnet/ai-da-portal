## 코드 스타일 가이드라인

아직 코드가 없으므로, 아래 규칙은 향후 작업을 위한 기본 가이드입니다. 이후 사용자
지시나 새로 추가된 프로젝트 파일이 다른 기준을 정하면 그 기준을 따릅니다.

### 일반 스타일

- 영리함보다 명확함을 우선할 것
- 요청을 완전히 해결하는 가장 작은 변경을 할 것
- 주변 코드가 생기면 그 스타일을 정확히 맞출 것
- 파일이 이미 Unicode 를 요구하지 않는 한 기본적으로 ASCII 를 사용할 것
- 불필요한 주석은 피하고, 의도가 자명하지 않을 때만 주석을 추가할 것
- 결정적인 동작과 명시적인 제어 흐름을 선호할 것

### 포매팅

- 각 파일의 기존 포매팅 관례를 유지할 것
- 포매터가 설정되지 않았다면 단순하고 일반적인 형식을 유지할 것
- 파일 내부 들여쓰기를 일관되게 유지하고, 관련 없는 코드를 재정렬하지 말 것
- 가독성을 위해 줄 길이는 적당히 유지하되, 임의의 길이 제한만 맞추기 위해 코드를
  불필요하게 흔들지 말 것
- 논리적으로 관련된 선언은 함께 묶을 것

### 임포트와 의존성

- 실제로 사용하는 것만 임포트할 것
- 파일을 수정할 때 죽은 임포트는 제거할 것
- 규칙이 생기면 절대 경로나 프로젝트 표준 임포트 경로를 선호할 것
- 임포트 순서는 로컬 파일이나 포매터 규칙과 일치시킬 것
- 명시적 승인 없이 새 외부 의존성을 도입하지 말 것
- 새 패키지보다 표준 라이브러리나 기존 유틸리티를 우선할 것

### 타입과 인터페이스

- 공개 경계에서는 명시적인 타입을 선호할 것
- 지나치게 넓은 만능 타입보다 가능한 한 좁은 타입을 사용할 것
- nullable 과 optional 은 의도를 가지고 모델링할 것
- 신뢰할 수 없는 입력은 경계에서 검증할 것
- 데이터 계약은 안정적이고 읽기 쉽게 유지할 것
- 타입 단언이나 캐스팅은 꼭 필요하고 정당화될 때만 사용할 것

### 네이밍

- 의도가 드러나는 설명적인 이름을 사용할 것
- 해당 도메인에서 표준인 경우가 아니라면 축약어보다 전체 단어를 선호할 것
- 주변 코드와 프레임워크 관례에 맞게 이름을 일관되게 유지할 것
- 단일 항목은 단수형, 컬렉션은 복수형 이름을 사용할 것
- 함수 이름은 구현 방식이 아니라 동작과 반환값을 기준으로 지을 것

### 함수와 모듈

- 함수는 하나의 책임에 집중하게 유지할 것
- 거대한 만능 헬퍼보다 조합을 선호할 것
- 가능한 경우 숨은 부작용보다 순수 함수를 선호할 것
- 모듈 API 는 작고 응집력 있게 유지할 것
- 필요가 입증되지 않았다면 깊은 추상화 계층을 만들지 말 것

### 에러 처리

- 디버깅에 충분할 만큼은 드러내되, 모호한 일반 에러는 피할 것
- 에러를 전파할 때 유용한 맥락을 보존할 것
- 예상 가능한 실패는 발생 지점 가까이에서 처리할 것
- 에러를 조용히 삼키지 말 것
- 언어나 프레임워크가 지원하면 구조화된 에러나 결과를 반환할 것
- 운영자나 개발자의 대응에 실제 도움이 될 때만 로깅을 추가할 것

### 테스트 기대사항

- 테스트 프레임워크가 생기면 동작 변경에 맞춰 테스트를 추가하거나 수정할 것
- 테스트는 검증하는 동작 가까이에 둘 것
- 같은 사실을 증명할 수 있다면 광범위하고 깨지기 쉬운 통합 테스트보다 목표가 분명한
  테스트를 선호할 것
- 테스트 이름은 관찰 가능한 동작 기준으로 지을 것
- 저렴하고 신뢰성 있게 실제로 실행할 수 있는 것은 과도하게 목킹하지 말 것

### 파일 및 변경 위생

- 작업에 실질적으로 도움이 되지 않는 한 파일 이름 변경이나 이동을 하지 말 것
- 관련 없는 영역에서 기회주의적 리팩터링을 하지 말 것
- 사용자가 커밋을 요청했을 때는 커밋 범위를 논리적으로 유지할 것
- 비밀 정보, 자격 증명, 환경 종속 산출물을 절대 커밋하지 말 것

### 리뷰 체크리스트

- 변경이 사용자 요청과 현재 저장소 상태에 맞는지 확인할 것
- 수정한 파일을 다시 읽어 포매팅 흔들림이나 미사용 임포트가 없는지 확인할 것
- 언급하려는 명령어가 실제로 존재하는지 확인한 뒤 말할 것
- 없는 도구를 추측하기보다 설정 공백을 명시적으로 설명할 것
- 검증 명령이 없다면 무엇을 확인했는지 대신 설명할 것
- 저장소가 발전하면 새 명령과 규칙을 이 파일에 기록할 것
